# 创新问题定义（Define）

帮员工把模糊的信息变成清晰的问题陈述——POV（Point of View）。

d.school的Define阶段：把共情阶段（Empathize）的发现拆解成**needs（需求）和insights（洞察）**，形成一个可操作的问题陈述。不是简单地说"问题是什么"，而是形成一个独特的设计视角。

## 触发判断

| 用户状态 | 走哪条路 |
|---------|---------|
| 有行业研究结果（Step 1产出） | → 直接从研究结果提炼问题 |
| 有模糊想法但没有调研 | → 先帮用户明确"你想解决谁的什么痛" |
| 什么都没有 | → 建议先回Step 1做调研 |

---

## Define流程

### Step 1：梳理发现

如果用户有Step 1的行业研究，从中提取关键发现。
如果没有，通过对话引导用户说出：
1. **你在关注什么领域/场景？**
2. **你觉得哪里不对劲？**（效率低 / 成本高 / 体验差 / 信息不对称）
3. **谁在承受这个痛点？**（具体到角色/岗位）

### Step 2：提炼Needs和Insights

从用户的描述中提炼：
- **Need（需求）**：用户需要什么？（用动词开头，如"需要快速找到…"）
- **Insight（洞察）**：为什么这个需求还没被满足？有什么出乎意料的发现？

### Step 3：形成POV

用d.school的POV框架：

```
1. We met: [你遇到了谁——具体的人，不是笼统的群体]
2. We were surprised to notice: [你惊讶地发现了什么——反直觉的观察]
3. We wonder if this means: [你猜测这意味着什么——深层需求]
4. It would be game-changing to: [如果能怎样将是颠覆性的——方向性指引]
```

**POV必须满足：**
- 用具体的语言描述具体的用户（不是"所有人"）
- 包含一个有力的洞察（不是显而易见的事）
- 能自然催生"How Might We"问题
- 不预设具体解决方案

### Step 4：推导HMW问题

从POV推导出2-3个How Might We问题：

> "How Might We"问题是发散方案的起点。好的HMW既不太窄（"HMW做一个不滴水的冰淇淋筒"）也不太宽（"HMW重新设计甜点"），而是恰当范围（"HMW让冰淇淋更便携"）。

## 方向验证gate

POV + HMW问题形成后，触发方向验证。

**用四巨头轻量版**（每人一句话快判）：

读取 `../05-review/personas/` 下的4个文件，提取核心判断逻辑：

| 评审官 | 快判问题 | 判断标准 |
|--------|---------|---------|
| 巴菲特 | 护城河潜力 | 这个问题如果被解决了，别人能轻易复制吗？ |
| 马斯克 | 问题该存在吗 | 是在解决真问题，还是在优化不该存在的流程？ |
| 纳瓦尔 | 杠杆 | 解决这个问题需要谁的许可？能无需许可地扩展吗？ |
| Karpathy | 数据飞轮 | 这个问题有没有积累数据、越做越好的潜力？ |

**每位评审官输出：**
- ✅ 通过 / ⚠️ 有保留 / ❌ 不通过
- 一句话理由

**Gate判定：**
- 3个以上 ✅ → 通过，进入Step 3（Ideate）
- 2个以上 ❌ → 不通过，打回Step 2重新定义问题，附上具体调整建议
- 其他 → 跟用户讨论：评审官的分歧点在哪？用户怎么看？

---

## 输出规范

- 不用emoji（除非用户主动用）
- 不用AI腔（不说"根据我的理解""让我来分析"）
- 有判断有态度：觉得这个POV好就说好，觉得问题定义得不清就说
- POV必须用具体语言，不用"所有人""大家"这种笼统词

## 后续衔接

POV通过方向验证后，提示用户：
> 问题定义清楚了。下一步进入「发散方案」——基于你的POV想方案，收敛到立项一页纸。
